# perspectipedia — Roadmap to Launch (v1 → public)

> תוכנן על ידי Claude Fable 5 (המודל החזק ביותר) · מול commit `234855f` · 2026-07-10
> זהו מסמך אסטרטגי חי. הביצוע בפועל נגזר ממנו.

## 0. איפה אנחנו

לולאת הליבה (`entryService.ts` → `claude.ts` → Prisma) נקייה ובעלת צורה נכונה:
cache-check → generate עם structured output → סינון מבחן הביסוס → persist → render.
הסכמה מקודדת באמת את ה-DNA העריכתי — `grounding` ו-`epistemic_type` הם שדות חובה.
**אבל אף אחד עדיין לא ראה את המערכת מייצרת ערך אמיתי.** כל מה שלמטה מסודר סביב העובדה הזו.

## 1. הסיכון הכי גדול: איכות פלט לא-מאומתת — לתקן קודם

תזת המוצר כולה היא "מבחן הביסוס מייצר עדשות נאמנות, לא-קריקטוריות, ומבוססות היטב".
כרגע זה נאכף על ידי ~40 שורות שבודקות רק ש**מחרוזות אינן ריקות**. עדשה שמבוססת על
מקור **מומצא** עוברת. זו עקב-האכילס של מבחן הביסוס: הוא מאמת את ה*צורה* של הביסוס, לא את ה*אמת*.

**לתקן עם eval harness, לא עם תחושות:**
1. **סט נושאי-זהב (~30)** על פני ספקטרום הסיכון: קנוני, אמפירי-גבולי, רגיש-למסורת, אדוורסרי, ושגרתי.
   לייצר פעם אחת עם מפתח אמיתי דרך **Batch API** (50% הנחה).
2. **רובריקה מנוקדת לכל עדשה, 5 ממדים (1-5):** דיוק ביסוס (האם המקור *קיים* ואומר מה שנטען),
   נאמנות/אנטי-קריקטורה, סימטריה, משמעת false-balance, בחירת עדשות.
3. **שתי שכבות מדידה:** LLM-as-judge (Opus 4.8 נפרד + web search לאימות מקורות),
   ומבחן ה"הייתי חותם" (3-5 אינסיידרים אמיתיים קוראים את העדשה שלהם).
4. **`prompts.ts` כארטיפקט מנוהל** — `PROMPT_VERSION`, נשמר על כל Entry, eval לפני כל שינוי.
   רף השקה מספרי: דיוק ביסוס ≥95% מהציטוטים ניתנים לאימות, אפס ציוני קריקטורה ≤2, ממוצע אינסיידר ≥4/5.
5. **תיקוני קוד זולים:** טיפול ב-`stop_reason: refusal` ו-`max_tokens`; לוג `usage`; שמירת הפלט הגולמי;
   `passesGroundingBar` חזק יותר (≥2 עדשות, אורך גוף מינימלי).

## 2. פריסה: Vercel + Neon Postgres

Vercel (Hobby→Pro) + Neon Postgres (free tier). Next.js 15 נפרס ב-Vercel בלי קונפיגורציה.
לעשות את מיגרציית ה-DB עכשיו כשאין נתוני production. מיגרציה ~30 דקות:
`provider="postgresql"`, `Lens.grounding String → Json` (מבטל את `JSON.parse/stringify`),
עמודות חדשות (usage, promptVersion, status), `DATABASE_URL` + `ANTHROPIC_API_KEY` ב-env.
**מינימום לפריסה = זה + רצפת העלויות (rate limit + מכסה יומית).** לא לפרוס בלי המכסה.

## 3. בעיית היצירה הארוכה: שורת-job + polling (v1.5)

`EntryView` מחזיק fetch יחיד פתוח לאורך כל היצירה (דקות) — שביר: proxies הורגים חיבורים,
refresh מאבד תשובה ומפעיל קריאת Opus שנייה, ואין dedup בין משתמשים.
**פתרון פרגמטי:** `status: pending|ready|failed` על Entry — השורה *היא* התור והמנעול
(unique slug = mutex). POST יוצר שורת pending ומחזיר 202; GET מחזיר סטטוס; client עושה polling.
עמיד ל-refresh, ל-multi-tab, ול-multi-user. **נדחה:** SSE/token streaming, מערכות job חיצוניות.

## 4. בטיחות ומודרציה: מבחן הביסוס הכרחי אך לא מספיק

**היכן הוא נכשל:**
1. **"מבוסס" ≠ "מקובל".** לאידיאולוגיות שנאה יש "מקורות ואסכולות" קוהרנטיים. מבחן הביסוס הוא
   מסנן אפיסטמי, לא מוסרי.
2. **לשון הרע** — ערך "אנציקלופדי" על אדם חי = חשיפה משפטית.
3. **ה-cache הופך כשל אחד לתוכן קבוע** — יצירה גרועה נשמרת ומוגשת לכולם.
4. **Drive-by generation** — ביקור בכל `/entry/<כלשהו>` מפעיל יצירה בתשלום ושמירה, בלי כוונת אדם.

**מעקות נדרשים לפני השקה:**
- **שער-נושא לפני יצירה:** קריאת Haiku 4.5 (~$0.001) המסווגת: `allow|refuse|needs_care`.
  סירובים נשמרים כשורות blocklist זולות. אדם חי → סירוב קטגורי ב-v1.
- **סעיף מדיניות ב-`SYSTEM_PROMPT`** + הרחבת הסכמה ב-`refused: boolean` — למודל חייבת להיות "לא" חוקית.
- **טיפול ב-`stop_reason: refusal`.**
- **חיסול drive-by:** ערכים קיימים ב-server-render; נושאים חדשים נוצרים רק מפעולה מפורשת.
- **רצפה תפעולית:** admin list+delete, לינק "דיווח", מבט יומי.

## 5. בקרת עלויות

~$0.15-0.40 לערך. הסיכון הוא abuse בנפח. בקרות, זול-קודם:
1. **מכסה יומית גלובלית (חובה, מהפריסה הראשונה):** hard-stop ב-`GEN_DAILY_CAP`.
2. **rate limit לפי IP:** Upstash Ratelimit.
3. **לוג usage** (טוקנים+עלות לערך).
4. **שיפור cache-hit:** נרמול נושאים + טבלת alias + בדיקת Haiku "האם תואם ערך קיים".
5. **כיוונון effort eval-gated:** לסרוק medium מול high; **להישאר על Opus 4.8** — הליבה רגישת-אינטליגנציה.

## 6. הצעד ההיברידי (v2): ה-DB כבר מקור האמת

הרינדור קורא משורות Prisma, לא מה-LLM — עריכה אנושית היא *תוספת*, לא כתיבה מחדש.
1. **עכשיו (מיגרציה אחת):** עמודות provenance — `promptVersion`, `model`, `editedByHuman`, `status`.
2. **הצעד האמיתי הקטן:** admin lens editor (route מוגן, textareas, badge "נבדק"). משמש גם לתיקון בטיחותי.
3. **אחר כך:** טבלת `LensRevision` (snapshots append-only) — diff/rollback, תשתית לעריכה קהילתית.

## 7. Roadmap מסודר

- **v1.1 — "להוכיח" (~שבוע, בלי פריסה):** מפתח אמיתי; סט זהב דרך Batch; רובריקה+judge+אינסיידרים;
  איטרציה על `prompts.ts`; חיזוק קוד (refusal/max_tokens/usage/validation/refused escape hatch); שער Haiku.
- **v1.2 — "פריסה בטוחה מינימלית" (~שבוע):** מיגרציית Postgres; Vercel; מכסה+rate limit; admin delete+דיווח;
  **server-render לערכים קיימים** (SEO); יצירה רק מפעולה מפורשת.
- **v1.5 — "UX מוצרי אמיתי" (2-3 שבועות):** job-row+polling; ספרייה+חיפוש; נרמול+alias; ליטוש השוואה;
  השקה ציבורית עם ~30 ערכים מעולים מראש.
- **v2 — "היברידי+אמון":** admin editor + `LensRevision` + badges; רינדור עדשה-אחר-עדשה; פאנל אינסיידרים.

**לא עכשיו:** חשבונות/קהילה (v3); fact-checking בזמן אמת; מובייל ייעודי; תשתית queue/vendor.
*רב-שפתיות — עודכן (D8, ENGINEERING_PLAN 4.9):* סכמת locale נכנסת כבר במיגרציית v3 (זול);
הפעלת אנגלית — רק אחרי שהעברית עוברת את ה-eval וקורפוס ההשקה חי.

### התזה במשפט
בשבועיים הקרובים להפוך את **האיכות למדידה** ואת **הכשל לחסום** (evals + מכסות + שער), כי כל
הצעת הערך של perspectipedia — מבחן הביסוס — היא כרגע הבטחה לא-בדוקה; כל השאר הוא הנדסת web רגילה
שהארכיטקטורה הקיימת כבר תומכת בה היטב.
